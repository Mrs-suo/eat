# 家庭成员模型改造方案

> 目标：把"做饭人 / 吃饭人"二元角色，改成**"家庭 + 今日主厨"**的轻量动态模型，匹配小两口"轮换做饭"的核心场景。

---

## 一、Why 优先级

### 业务问题（用户视角）
- **强迫承诺**："一注册就选做饭/吃饭"对小两口不真实。两人经常轮换做，谁也不是专职。
- **绑定倒挂**：现在做饭人"邀请"吃饭人，但小两口是平等关系，不是服务关系。
- **切换成本**：今天我做菜就得切账号，体验差。
- **数据空洞**：用户每次都要想"我今天是哪种人"，心智负担高。

### 业务目标（产品视角）
- 降低首次使用门槛（**注册转化率 +20%**）
- 提升"今日"页停留与互动（**菜品发布/点餐频次 +30%**）
- 沉淀**家庭**这个核心实体（为后续"家庭共享菜谱、家庭账本、家庭相册"等扩展打基础）

---

## 二、产品定位调整

**Before**：个人工具型 App（"我是做饭人，我邀请你吃饭"）
**After**：家庭工具型 App（"我们这个家，今天你做明天我做"）

品牌文案建议调整：
- "吃什么" → "**家厨** · 今天吃啥"
- 登录页副标题：登录后加入或创建你的家庭

---

## 三、核心数据模型

### 3.1 新增实体（最小改动原则）

| 实体 | 字段 | 说明 |
|------|------|------|
| `family` | id, family_code (6位), name, create_time | 家庭 |
| `family_member` | user_id, family_id, nickname, role, joined_at | 家庭成员（多对多） |
| `daily_cook` | id, family_id, cook_user_id, cook_date | 每日主厨 |

### 3.2 `app_users` 表的兼容改动

为了**平滑迁移**，保留原字段，新增可选字段：

```sql
ALTER TABLE app_users
  ADD COLUMN nickname VARCHAR(20) DEFAULT NULL COMMENT '家庭内昵称',
  ADD COLUMN default_role VARCHAR(20) DEFAULT NULL COMMENT '默认偏好 cook/eat/null';
```

**兼容策略**：
- 老用户：`role=cook` 的默认偏好=cook；`role=eat` 的默认偏好=eat
- 老绑定关系 `boundCookUserId` 暂时保留，作为"今日主厨"的兜底

### 3.3 关系图

```
Family (1) ─── (N) FamilyMember ─── (1) AppUser
   │
   └── (N) DailyCook (按日期一条记录)
```

## 四、用户故事 & 关键流程

### 4.1 注册流程（新）

**作为** 一位新用户，
**我希望** 输入手机号后直接进入"创建/加入家庭"页面，
**以便** 不被"我是做饭还是吃饭"这种承诺卡住。

```
[输入手机号] → [选择身份]
              ├─ 创建家庭（输入家庭名、我的昵称）→ 成为家庭第一个成员
              └─ 加入家庭（输入 6 位家庭码）→ 加入对方家庭
```

> **关键决策点**：把"选角色"挪到家庭创建完成**之后**的个人设置里，且允许随时修改。

### 4.2 今日主厨流程

**作为** 一个家庭成员，
**我希望** 在"今日"页顶部一眼看到"今天是谁做饭"，
**以便** 知道点什么菜、谁负责下厨。

```
[进入今日页] → 显示今日主厨头像 + 昵称
            → 非主厨：底部"+ 我来做今天的饭"按钮
            → 主厨：底部"+ 发布今日菜品"按钮
```

### 4.3 切换主厨流程

**作为** 临时想换人做饭的成员，
**我希望** 长按主厨头像可以发起"换人做饭"，
**以便** 不需要打电话就能通知对方。

```
[长按今日主厨头像] → 弹出"换 TA 来做"或"换我来"
                  → 一键切换今日主厨（仅当日有效）
```

### 4.4 历史菜品归属

**作为** 想回顾上周谁做了什么的成员，
**我希望** 历史菜品的发布人显示昵称 + 头像，
**以便** 知道"那天的红烧肉是他做的"。

> **实现**：发布菜品时记录 `cookUserId`，展示时通过家庭成员映射到昵称。

## 五、UI/UX 改造点

### 5.1 登录注册页（立即可做）

**Before**：
- 注册 tab 下强制显示"做饭账号 / 吃饭账号"两个大卡片

**After**：
- 删掉角色选择卡片
- 替换为一句温柔的引导文案：
  > 一个人用 → 直接创建你的家庭
  > 两个人用 → 一个人先注册，把家庭码发给另一个人
- 登录按钮下方新增两个 tab：**「创建家庭」/「加入家庭」**（注册时显示）

### 5.2 今日页（核心改造）

**新增顶部"今日主厨"卡片**：

```
┌─────────────────────────────────┐
│  👨‍🍳 今天主厨  ▸                 │
│                                 │
│  ┌───┐  小明                    │
│  │👤 │  做饭偏好：红烧 / 川菜     │
│  └───┘                          │
│                                 │
│  [换 TA 来做]   [换 我来做]      │
└─────────────────────────────────┘
```

### 5.3 我的页（兼容老用户）

**保持现有"做饭/吃饭"展示**（避免老用户困惑），但新增：
- 家庭信息卡（家庭码、成员列表）
- "今日我是：主厨 / 食客"实时状态
- 切换身份的快捷入口

## 六、API 设计（增量）

### 新增接口

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST /api/families` | 创建家庭 | 返回 `familyId` + `familyCode` |
| `GET /api/families/{code}` | 查询家庭信息（加入前预览） | |
| `POST /api/families/{code}/join` | 加入家庭 | |
| `GET /api/families/{id}/members` | 家庭成员列表 | |
| `GET /api/families/{id}/today-cook` | 今日主厨 | |
| `POST /api/families/{id}/switch-cook` | 切换今日主厨 | body: `{userId, date}` |

### 兼容接口

- `POST /api/users/register` → 新增可选参数 `familyCode`、`nickname`
- 老 `boundCookUserId` 字段保留，**逐步废弃**（不主动删除，避免数据丢失）

## 七、数据迁移方案

### 7.1 老用户自动迁移
- 启动时跑一次数据脚本：
  ```sql
  -- 给老用户创建一个"个人家庭"
  INSERT INTO family (family_code, name, create_time)
  SELECT CONCAT('LEGACY', LPAD(id, 4, '0')), CONCAT(role, '_', phone), NOW()
  FROM app_users WHERE nickname IS NULL;

  -- 关联成员
  INSERT INTO family_member (user_id, family_id, nickname, role, joined_at)
  SELECT user_id, family_id, phone, role, create_time FROM ...;
  ```

### 7.2 绑定关系兼容
- 老的 `boundCookUserId` 关系 → 在新模型里转为"家庭内成员关系"
- 提示老用户在"我的"页"升级为家庭模式"（一次性的 banner）

## 八、指标体系

### 北极星指标
**家庭周活跃数**（WAU of Family）：每周至少 1 次主厨切换或菜品发布的家庭数

### 驱动指标
| 指标 | 基线 | 3 个月目标 |
|------|------|-----------|
| 注册转化率 | 60% | 80% |
| 双人家庭占比 | 0% | 30% |
| 今日主厨切换频次 | 0/周 | 2次/周 |
| 今日页停留时长 | 30s | 90s |

### 健康指标（防过度）
- 家庭平均成员数（防止"拉人刷数据"）
- 主厨连续天数（防止"一个人一直做"的负面体验）
- 家庭退出率

## 十、风险与边界

### Non-goals（明确不做）
- ❌ 不做多家庭切换（一个用户只属于一个家庭）
- ❌ 不做"家庭管理员"权限分级（家庭成员平等）
- ❌ 不做实时聊天（用 IM 工具替代）
- ❌ 不做跨家庭菜品分享（先聚焦单家庭场景）

### 风险
| 风险 | 应对 |
|------|------|
| 老用户不愿意升级 | 保留老模式入口，半年后强制升级 |
| 家庭码泄漏 | 6 位数字码 + 24h 加入链接变体 |
| 主厨切换被滥用 | 加"每周最多切换 3 次"软限制 |
| 数据迁移失败 | 保留原表半年，写脚本回滚预案 |
